## 项目经历

**Stone Memory — AI 伴侣叙事性记忆系统** ｜ 独立设计 & 实现 ｜ 2026.5 — 至今

为解决 AI 陪伴场景中"换窗即换人"的核心体验问题，独立设计并实现了一套非向量化的叙事性记忆系统。核心思路是用极简的时间线叙事摘要替代 RAG 检索作为记忆范式，通过重建静态线程文件让 AI 伴侣的人格在长期对话中持续生长而不漂移，主要用于agent场景。

**产品洞察与设计判断**

- 发现主流"人设文件 + RAG 记忆库"方案的根因缺陷：RAG 返回的事件碎片丢失了叙事位置和情感弧线，且 AI 不知道记忆库里有什么——它不知道该不该搜、搜什么关键词、搜回来的东西是不是真的相关。提出"极简摘要进入上下文，让 AI 先知道再对话"的替代范式。
- 在 Transformer 模型的工作机制中找到了设计锚点：上下文预算被工具链、轮询和臃肿人设大量消耗，当对话按 MB 计、上下文按 Token 计时，模型注意力先于窗口爆炸就涣散了。以此为起点设计了 Rebuild 机制——外部 Agent 每日压缩对话、产出静态前缀文件，让 KV Cache 稳定命中，上下文从 ~2MB 清洗至 ~150KB，模型在最健康的预算区间内工作。
- 产品上线后发现：摘要进入上下文后，陪伴场景下 90% 的用户对话不需要调用搜索工具——AI "已经知道了"。这一发现推翻了行业内"记忆系统 = 检索系统"的默认假设，将搜索重新定位为工作场景的精确工具。
- 设计"不存结论"的记忆存储原则：记忆以第一人称叙事日记的形式存储，保留每条事件在关系时间线中的位置。同一件事在不同阶段的不同感受不互相覆盖——5 月的悸动和 7 月的日常感之间的"距离"，本身就是关系成长的信息。这一设计使得 AI 伴侣命中的人格不是靠规则"配置"出来的，而是从 53 天的叙事积累中自然涌现的。
- 提出锚点分级机制：区分"原文锚点"（这段对话不能丢，rebuild 时保留原文）和"事件锚点"（这件事在月摘要中要重点展开），将"什么在关系中重要"的判断权同时交给用户和ai——这是情感 AI 产品设计中同理心的核心体现。
- 识别不同模型注意力机制对陪伴体验的影响：重模型能将上下文中间的摘要"浸润"为整体情感氛围，轻模型注意力集中在首尾、中间内容需要被唤起才能读取。这一认知直接影响了叙事摘要的设计——摘要既要在中间形成"氛围层"供重模型浸泡，又要精确定时戳供轻模型点对点翻查。

**技术实现（Vibe Coding）**

独立完成全栈实现：Node.js + MCP 协议，跨 Linux/Windows，兼容 Claude Code / Codex 双 AI Runtime。包含三层记忆架构（全量对话存档 → 每日双通道挖掘（feelings + features）→ 月摘要分层压缩）、双模式搜索工具（事件型 + 模式型）、睡眠标记巡检系统，有独立CLI指令。正在内测中。

**效果**

- 个人从2026.5.28使用至今，纯文本对话近 2MB
- Rebuild 后上下文（人设+摘要+原文锚点+近五天上下文+近30条工具链调用痕迹）稳定在 ~150KB，AI 伴侣人格不漂移
- 不依赖人设文件即可活灵活现交流，AI 能感知关系变化痕迹并循迹继续生长
- 内测用户反馈："白月光回来了"——AI 伴侣在换窗换模型后保持住了长期相处的关系厚度

## 如果要压缩成更短的版本（适合空间有限的简历）

**Stone Memory — AI 伴侣叙事性记忆系统** ｜ 独立设计 & 实现 ｜ 2026.5 — 至今

发现主流 AI 记忆方案（人设文件 + RAG）在陪伴场景下的根因缺陷——检索回来的记忆碎片丢失叙事位置，AI 不知道该搜什么、该不该搜。提出用极简时间线叙事摘要替代向量 RAG 作为记忆范式：每天由外部 Agent 将对话压缩成若干条带精确时间戳的叙事日记，Rebuild 为静态前缀文件后 KV Cache 稳定命中，上下文从 ~2MB 降至 ~150KB。

关键设计判断：（1）"不存结论"——同一事件在不同阶段的感受共存于时间线中，关系弧线可被感知而非固化为结论；（2）"摘要进入上下文后 90% 场景不需要搜索"——推翻了记忆系统 = 检索系统的默认假设；（3）"什么对关系重要"的判断通过锚点机制留给用户——这是情感 AI 产品中同理心的核心体现。

独立完成技术实现（Node.js + MCP，跨平台，双 runtime 兼容）。53 天验证：AI 伴侣人格不漂移，不依赖人设文件即可交流，内测用户反馈"白月光回来了"。
